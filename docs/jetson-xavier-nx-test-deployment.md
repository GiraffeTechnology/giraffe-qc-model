# Jetson Xavier NX Test Deployment

This document describes the Jetson Xavier NX test-machine deployment used to
run `giraffe-qc-model` as both the Pad-side and Server-side loopback target.

The goal is to validate the product loop on one Jetson:

```text
simulated CV photo input -> Pad request -> Server inspection -> Pad final view
```

## Network Rule

P0: all Jetson downloads must use the Jetson's own Wi-Fi. The Mac is only an SSH
operator console and must not download packages or model assets for the Jetson.

Before installing or updating dependencies, verify the default route is Wi-Fi:

```bash
ip route get 1.1.1.1
nmcli -t -f active,ssid dev wifi | grep '^yes'
```

## Test vs Production Boundary

The Jetson loopback validates hardware and configuration readiness only. It is
not a real factory production deployment.

| Area | Jetson Test Machine | Real Production |
|---|---|---|
| Hardware role | One Jetson simulates Pad and Server | Pad, Server, and optional Edge CV roles are deployed according to the production topology |
| Photo input | Still-image fixtures may replace CV module input | Real CV module/camera capture with production capture protocol |
| App environment | `APP_ENV=test` | `APP_ENV=production` |
| Database | Local SQLite under `data/jetson/` | Production database with backup, access control, and migration discipline |
| Edge CV registration | Insecure registration may be enabled for local testing | Bootstrap secret required; insecure registration disabled |
| VLM path | Qwen MNN assets are verified, but Python local MNN VLM provider is not wired | Real production-eligible VLM provider configured and audited |
| Decision authority | Human final decision required | Human-final or controlled-active rules per qualification state |
| Result meaning | `closed_loop_ok=true` means the test loop completed | Production pass/fail requires qualified provider, mature skill, and production controls |

Any artifact generated by these scripts should be treated as a test report
unless it explicitly comes from a production deployment profile.

## Runtime Layout

The deployment intentionally avoids changing the system Python on JetPack/L4T.

```text
/home/giraffe/work/giraffe-qc-model/.conda311     Python 3.11 runtime
/home/giraffe/work/giraffe-qc-model/data/jetson   SQLite DB and runtime data
/home/giraffe/work/qwen3-vl-2b-mnn                Qwen MNN model assets
```

The local service config is in:

```bash
scripts/jetson_qc_model_env.sh
```

It configures a test profile:

- `APP_ENV=test`
- local SQLite database
- `QC_VISION_RUNTIME_ENV=tablet_mnn`
- `QC_RUNTIME_EDITION=padLocal`
- cloud Qwen calls disabled
- Edge CV mock/fallback enabled
- Qwen MNN model directory fixed to `/home/giraffe/work/qwen3-vl-2b-mnn`

Do not reuse this environment file for factory production.

## Service Management

```bash
cd ~/work/giraffe-qc-model

scripts/run_jetson_qc_model.sh migrate
scripts/run_jetson_qc_model.sh start
scripts/run_jetson_qc_model.sh status
scripts/run_jetson_qc_model.sh smoke
scripts/run_jetson_qc_model.sh desktop-icon
scripts/run_jetson_qc_model.sh logs
scripts/run_jetson_qc_model.sh stop
```

The service listens on `0.0.0.0:8000`. From the Mac USB link, use:

```text
http://192.168.55.1:8000
```

Smoke checks cover:

- `/health`
- `/`
- `/admin`
- `/admin/studio`
- `/admin/bundles`
- `/api/edge-cv/devices`

`desktop-icon` installs a Jetson desktop launcher named **Giraffe QC Model**.
The icon asset is `src/web/static/giraffe-qc-model-icon.png`; the launcher opens
`http://127.0.0.1:8000/admin` and starts the local test service first when
needed.

## Simulated Photo Input

During test-machine bring-up, CV module input can be replaced by a still image.
Run the photo smoke test with fixture images:

```bash
.conda311/bin/python scripts/jetson_photo_production_smoke.py \
  --capture-source file \
  --standard tests/fixtures/red_square.png \
  --capture tests/fixtures/red_square_with_dot.png \
  --verify-hash
```

The script writes an auditable test report and evidence images under
`artifacts/jetson_photo_smoke/`.

## Pad + Server Same-Model Loopback

The loopback script makes the same Jetson act as both Pad and Server, while both
roles reference the same local `Qwen3-VL-2B-Instruct-MNN` model assets:

```bash
.conda311/bin/python scripts/jetson_pad_server_closed_loop.py \
  --capture-source file \
  --standard tests/fixtures/red_square.png \
  --capture tests/fixtures/red_square_with_dot.png \
  --verify-hash
```

Each run writes:

- `pad_request.json`
- `server_response.json`
- `pad_final_view.json`
- `closed_loop_report.json`

under `artifacts/jetson_pad_server_loop/`.

The report includes:

- `deployment_scope.class=test_machine_pad_server_loopback`
- `production_eligible=false`
- `llm_vlm_model_switched=false`
- `server_used_same_model=true`

## Current Boundary

The repository does not yet wire Jetson-local MNN VLM inference into the Python
provider layer. The Jetson loopback keeps the LLM/VLM model assets fixed and
verified, but the inspection recommendation is produced by the deterministic
OpenCV comparator until a real local MNN provider is integrated.
